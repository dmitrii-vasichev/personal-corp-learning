# PRD: Дедлайны задач
**Проект:** Онкошкола Портал (Task Manager)
**Версия:** 1.0
**Статус:** Draft

---

## 1. Проблема

Сейчас задачи в системе существуют без временных рамок. Это приводит к тому, что:

- Исполнители не знают, когда именно нужно сдать задачу — срочность определяется только на словах в чате
- Настя и другие координаторы вынуждены вручную контролировать сроки и напоминать о них в Telegram-чате
- Задачи из итогов встреч (Zoom-суммаризации) часто содержат явные сроки («до пятницы», «до 15-го»), но система их игнорирует и не фиксирует
- Нет единого места, где видно «что просрочено» — приходится держать это в голове или в отдельных таблицах

**Итог:** команда из 20 человек теряет время на ручное напоминание и управление сроками вместо того, чтобы делать работу.

---

## 2. Цель

После реализации фичи:

- Каждая задача может иметь дедлайн (дата + опционально время)
- Telegram-бот автоматически напоминает исполнителю за 24 часа и в день дедлайна
- Просроченные задачи визуально выделяются в веб-интерфейсе
- Парсер Zoom-транскрипций распознаёт упоминания сроков и проставляет дедлайн автоматически
- Координатор видит в одном месте все задачи команды с дедлайнами, отсортированные по срочности

---

## 3. Пользовательские сценарии

**Сценарий 1 — Координатор создаёт задачу со сроком (веб)**
Настя открывает веб-интерфейс, создаёт задачу «Подготовить материалы к вебинару» и выбирает дедлайн «15 марта 18:00». Задача появляется у исполнителя в Telegram с указанием срока. За 24 часа до дедлайна исполнитель получает напоминание.

**Сценарий 2 — Парсинг дедлайна из Zoom-итогов**
Настя вставляет в бот транскрипцию встречи, где написано «Иван — сделать презентацию до пятницы». Система создаёт задачу и автоматически вычисляет дату ближайшей пятницы как дедлайн. Иван получает уведомление с указанием срока.

**Сценарий 3 — Исполнитель проверяет свои задачи в Telegram**
Иван пишет `/tasks` в бот. Бот возвращает список его задач, отсортированных по дедлайну: сначала просроченные (🔴), затем с ближайшими сроками (🟡), затем без срока.

**Сценарий 4 — Просроченная задача**
Дедлайн по задаче прошёл, а статус не изменён. Задача помечается как просроченная (overdue). Координатор видит это на доске задач. Исполнитель получает уведомление «Задача X просрочена».

## 4. Функциональные требования

### 4.1 Модель задачи
- Поле `deadline: datetime | null` — дата и время дедлайна, опционально
- Поле `deadline_notified_24h: bool` — флаг отправки напоминания за 24 часа
- Поле `deadline_notified_overdue: bool` — флаг отправки уведомления о просрочке

### 4.2 Создание/редактирование дедлайна
- При создании задачи через веб-интерфейс — датапикер для выбора дедлайна (необязательный)
- При создании задачи через Telegram-бот — возможность указать срок текстом: «/new задача | 15 марта» или «/new задача | завтра 18:00»
- Редактирование дедлайна существующей задачи через веб и через бот (`/deadline <task_id> <date>`)

### 4.3 Напоминания (Telegram)
- За 24 часа до дедлайна — уведомление исполнителю
- В момент наступления дедлайна (или через 1 час после) — уведомление о просрочке если задача не закрыта
- Все уведомления — личные сообщения исполнителю, не в общий чат
- Фоновый scheduler (APScheduler или celery beat) проверяет дедлайны каждые 15 минут

### 4.4 Отображение в веб-интерфейсе
- На карточке задачи отображается дедлайн: дата и время
- Цветовая индикация:
  - 🔴 Красный — просрочена (deadline < now, статус не done)
  - 🟡 Жёлтый — истекает скоро (deadline < now + 24h)
  - ⚪ Без цвета — дедлайн в норме или не задан
- Сортировка списка/доски: опция «по дедлайну»
- Фильтр «Просроченные» в представлении команды

### 4.5 Парсинг дедлайна из текста
- При создании задачи через AI-парсинг транскрипций — распознавание сроков: «до пятницы», «до 15-го», «завтра», «через неделю», конкретные даты
- Распознанный дедлайн проставляется автоматически, координатор видит его в превью перед подтверждением

---

## 5. Вне scope

- Повторяющиеся задачи (recurring deadlines) — не делаем
- Дедлайны для подзадач / чеклистов — не делаем
- Интеграция с Google Calendar / Outlook — не делаем
- Публичные дедлайны (видимые клиентам) — не делаем
- Изменение дедлайна через голосовое сообщение — не делаем (только создание через голос)
- SLA-метрики и отчёты по соблюдению дедлайнов — не делаем в этой версии
- Напоминания по email — не делаем (только Telegram)

## 6. Технический дизайн

### 6.1 База данных (Supabase / PostgreSQL)

```sql
-- Миграция: добавить поля в таблицу tasks
ALTER TABLE tasks
  ADD COLUMN deadline TIMESTAMPTZ DEFAULT NULL,
  ADD COLUMN deadline_notified_24h BOOLEAN DEFAULT FALSE,
  ADD COLUMN deadline_notified_overdue BOOLEAN DEFAULT FALSE;

CREATE INDEX idx_tasks_deadline ON tasks(deadline) WHERE deadline IS NOT NULL;
```

### 6.2 API (FastAPI)

```
PATCH /api/tasks/{task_id}
  body: { deadline: "2025-03-15T18:00:00Z" }
  → обновляет дедлайн, сбрасывает флаги уведомлений

GET /api/tasks?filter=overdue
  → возвращает задачи где deadline < now AND status != done

GET /api/tasks?sort=deadline
  → сортировка по дедлайну (nulls last)
```

### 6.3 Scheduler (Backend)

```python
# APScheduler job, каждые 15 минут
async def check_deadlines():
    # 1. Найти задачи где deadline < now + 24h и не notified_24h
    #    → отправить Telegram уведомление, поставить флаг
    # 2. Найти задачи где deadline < now и не notified_overdue и status != done
    #    → отправить Telegram уведомление, поставить флаг
```

### 6.4 Telegram-бот

Новые команды:
- `/deadline <task_id> <дата>` — установить/изменить дедлайн
- `/tasks` — обновить вывод: добавить дедлайн и эмодзи-индикатор к каждой задаче

Парсинг даты из текста: библиотека `dateparser` (поддерживает русский язык).

### 6.5 Веб-интерфейс (Next.js)

- Компонент `DeadlinePicker` — обёртка над `react-datepicker` или `shadcn/ui Calendar`
- Компонент `DeadlineBadge` — цветной бейдж с датой (логика: overdue / soon / normal)
- Обновить `TaskCard` и `TaskList` — добавить `DeadlineBadge`
- Фильтр `overdue` в `TeamTasksView`

## 7. Метрики успеха

Фича считается готовой, если выполняются все пункты:

- [ ] Создать задачу с дедлайном через веб → дедлайн сохраняется в БД и отображается на карточке
- [ ] Создать задачу с дедлайном через Telegram (`/new задача | 15 марта`) → дедлайн парсится и сохраняется
- [ ] Вставить транскрипцию с текстом «до пятницы» → задача создаётся с автоматически вычисленным дедлайном
- [ ] Подождать/сэмулировать наступление дедлайна − 24h → исполнитель получает Telegram-уведомление
- [ ] Дедлайн прошёл, задача не закрыта → задача получает статус overdue, карточка красная в веб
- [ ] `/tasks` в Telegram показывает дедлайны и сортирует: просроченные первыми
- [ ] Фильтр «Просроченные» в веб-интерфейсе показывает только overdue задачи
- [ ] Изменить дедлайн через веб → флаги уведомлений сбрасываются, новые уведомления будут отправлены
