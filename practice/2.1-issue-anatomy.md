# Issue Anatomy Cheatsheet

Краткое руководство по созданию качественных Issues в GitHub.

---

## Title

Заголовок должен кратко и конкретно описывать суть задачи.

| | Пример |
|---|---|
| **Плохо** | `Fix bug` |
| **Плохо** | `Update something` |
| **Хорошо** | `fix: login fails when email contains uppercase letters` |
| **Хорошо** | `feat: Add dark mode toggle to settings page` |
| **Хорошо** | `docs: Add API authentication guide to README` |

**Правила для хорошего заголовка:**
- Начинается с префикса типа (`feat:`, `fix:`, `docs:`, `chore:`)
- Описывает что именно нужно сделать или что сломано
- Не превышает ~70 символов
- Понятен без дополнительного контекста

---

## Description

Описание объясняет контекст задачи — зачем это нужно и что произойдёт.

**Шаблон:**
```
Краткое описание задачи в 2-3 предложениях.
Объяснение, почему это важно или что это решает.
```

---

## Acceptance Criteria

Список конкретных, проверяемых условий завершения задачи. Каждый критерий можно проверить вручную или автотестом.

**Примеры проверяемых критериев:**

- [ ] Кнопка "Войти" деактивирована при пустом поле email
- [ ] При неверном пароле отображается сообщение "Неверный email или пароль"
- [ ] Файл `practice/2.1-issue-anatomy.md` создан и содержит все обязательные секции
- [ ] Таблица labels содержит минимум 6 записей с описаниями

**Признаки хорошего критерия:**
- Начинается с глагола действия или с описания наблюдаемого результата
- Однозначно понятно — выполнен критерий или нет
- Не содержит слов "хорошо", "правильно", "корректно" без уточнения

---

## Technical Notes

Технические детали реализации: какие файлы трогать, какой формат использовать, ограничения.

**Примеры технических заметок:**

- Файл: `practice/2.1-issue-anatomy.md`
- Формат: Markdown
- Не использовать сторонние библиотеки — только стандартный `fetch` API
- Изменения только в папке `src/components/`, не трогать `src/utils/`
- Придерживаться существующего стиля именования: `camelCase` для переменных, `PascalCase` для компонентов

---

## Labels

Стандартные GitHub labels для категоризации Issues и PR.

| Label | Описание |
|---|---|
| `bug` | Что-то работает не так, как ожидается |
| `feat` / `enhancement` | Новая функциональность или улучшение существующей |
| `documentation` | Изменения касаются только документации |
| `chore` | Техническая задача без изменения функциональности (обновление зависимостей, конфиги) |
| `refactor` | Переработка кода без изменения поведения |
| `test` | Добавление или исправление тестов |
| `question` | Вопрос, требующий обсуждения |
| `wontfix` | Задача не будет выполнена (с объяснением в комментарии) |
| `good first issue` | Подходит для новых участников проекта |

---

## Полный шаблон Issue

```markdown
## Description
[Что нужно сделать и зачем]

## Acceptance Criteria
- [ ] [Критерий 1]
- [ ] [Критерий 2]

## Technical Notes
- [Технические детали, ограничения, файлы]
```
